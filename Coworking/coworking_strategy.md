# Coworking Referral Strategy for Rafi

## The Problem

Tandem occasionally receives mates who need private space for 2–3 people but aren't ready for a multi-year lease. Today there is no structured way to handle them: demand ops (Mariel, Sandra) manually chase information, and when Tandem doesn't have a matching listing they tell the client to go check out WeWork or Industrious directly — leaving the fee and the relationship on the table.

The goal is an 80/20 solution: enough coverage and speed to serve clients well, earn a referral fee, and keep the relationship for future private office conversion.

**What Tandem already has:**
- Some boutique coworking centers listed in the DB (`leaseType = 'COWORKING'`), but no systematic coverage of major operators
- Basecamp (BAM/BH in SF) is Tandem-operated — separate from this initiative
- No broker registration with any major coworking operator

**Key constraint:** Demand ops needs to respond to a mate quickly. The solution must front-load the research so Mariel/Sandra can act in minutes, not hours.

## The Approach: One Agent, Three Modes

Inventory comes entirely from operator correspondence -- there is no web crawling. `coworking_agent.py` handles three tasks: parsing what operators send back, drafting availability requests for operators with no regular cadence, and drafting inquiry emails when a mate comes in.

### Mode 1: `--ingest` (supply backlog, run when an operator responds)

Parses an operator's availability email from stdin and updates `coworking_locations.csv` with current pricing, suite availability, and touring links. Supply backlog's role: receive the email, pipe it in, verify the sheet updated.

```
cat operator_email.txt | python coworking_agent.py --ingest --operator "Werqwise" --market SF
cat operator_email.txt | python coworking_agent.py --ingest --operator "Werqwise" --market SF --dry-run
```

Each operator row in the CSV gets a `last_updated` date. Rows not updated in over 14 days are flagged as stale -- demand ops sees this before sending options to a mate.

### Mode 2: `--refresh` (supply backlog, ad hoc)

Drafts a fresh availability request to one operator or all operators, for cases where no regular cadence was established or data has gone stale. Routes the draft to `DEMAND_OPS_EMAIL` for review before forwarding.

```
python coworking_agent.py --refresh --operator "Mindspace" --market SF
python coworking_agent.py --refresh           # all operators with no recent update
python coworking_agent.py --refresh --dry-run
```

**Sample email subject:** `Mindspace San Francisco -- Availability + Touring Links | Tandem June 2026`

### Mode 3: `--mate` (demand matchmaking, on-demand)

Demand ops runs this when a mate comes in. It reads the CSV, finds locations matching the market and budget, and drafts a broker inquiry email for each match -- pre-filled with the client's profile, a request for availability confirmation, and a touring link to share. Flags any stale locations so the team knows to verify before sending.

```
python coworking_agent.py --mate "Acme Co" --headcount 3 --market SF --budget 4000
python coworking_agent.py --mate "Acme Co" --headcount 3 --market SF --budget 4000 --send
```

`--send` routes the drafts to `DEMAND_OPS_EMAIL` for one-click review before forwarding to the operator. Time from client inquiry to sent email: under 5 minutes.

### What the agent does vs. does not do

| | |
|---|---|
| Parses operator response emails into structured CSV rows | Drafts broker inquiry emails tailored to each client profile |
| Drafts availability request emails when data goes stale | Flags stale inventory so demand ops knows to verify before sending |
| Routes all drafts to demand ops for review before sending | Has real-time availability -- no operator has an open API for this |
| Handles any operator format -- emails, tables, PDFs pasted as text | Replaces the broker registration step (must be done once, manually) |

## Team Ownership and Process Flow

Coworking does not fit the standard supply or demand workflow. Supply cannot list individual coworking suites as stable inventory -- availability changes daily. Demand cannot chase inventory on the fly -- mates move too fast. The split: supply owns operator relationships and the inventory feed; demand ops consumes a maintained reference alongside standard listings.

Coworking options are not a separate track. They are part of the same matchmaking output as private office listings, including existing coworking spaces already in the DB (`leaseType = 'COWORKING'`).
